# 阶段5：原始数据清洗与存储实现说明

## 1. 目标

阶段 5 的目标是把阶段 4 采集到的结构化结果，正式落入业务表，而不是只保存在任务摘要里。

本阶段完成后，系统已经具备以下能力：

- 将视频基础信息写入 `video`
- 将任务上下文里的搜索排名与综合评分写入 `task_video`
- 将当前任务采集到的互动指标快照写入 `video_metric_snapshot`
- 将清洗后的简介、字幕和组合文本写入 `video_text_content`
- 将字幕分段写入 `video_subtitle_segment`
- 在同一任务重复执行时，先清理旧的任务产物，再重新写入，避免重复污染

## 2. 本阶段落地内容

### 2.1 数据落库服务

新增服务：

- [video_storage_service.py](E:\code\fullstack\spiderbilibili\backend\app\services\video_storage_service.py)

该服务负责：

- 重置某个任务已有的阶段性产物
- 按 `bvid` 更新或创建 `video`
- 按 `task_id + video_id` 更新或创建 `task_video`
- 按 `task_id + video_id` 更新当前任务的最新指标快照
- 按 `task_id + video_id` 更新或创建清洗后的文本内容
- 重建当前文本内容对应的字幕分段

### 2.2 文本清洗服务

新增服务：

- [text_clean_service.py](E:\code\fullstack\spiderbilibili\backend\app\services\text_clean_service.py)

当前清洗规则包括：

- 统一换行符
- 压缩多余空白字符
- 压缩连续空行
- 保留简介和字幕两个来源
- 生成用于 AI 分析的 `combined_text`
- 生成 `combined_text_hash`
- 当简介和字幕都为空时，使用标题作为兜底文本
- 当组合文本过长时进行截断

## 3. 数据写入映射

### 3.1 `video`

来源：视频详情接口

主要写入字段：

- `bvid`
- `aid`
- `title`
- `url`
- `author_name`
- `author_mid`
- `cover_url`
- `description`
- `tags`
- `published_at`
- `duration_seconds`

说明：

- `video` 是全局视频主表
- 相同 `bvid` 会更新，而不是重复插入

### 3.2 `task_video`

来源：任务上下文 + 综合评分结果

主要写入字段：

- `task_id`
- `video_id`
- `search_rank`
- `keyword_hit_title`
- `keyword_hit_description`
- `keyword_hit_tags`
- `relevance_score`
- `heat_score`
- `composite_score`
- `is_selected`

说明：

- 该表表达“某个任务里，这个视频是如何被选中和评分的”
- 同一个视频在不同任务下可以有不同的搜索排名和评分

### 3.3 `video_metric_snapshot`

来源：视频详情接口的互动指标

主要写入字段：

- `view_count`
- `like_count`
- `coin_count`
- `favorite_count`
- `share_count`
- `reply_count`
- `danmaku_count`
- `captured_at`
- `metrics_payload`

说明：

- 当前阶段采用“同一任务保留一份最新快照”的策略
- `metrics_payload` 会保留搜索侧指标、详情侧指标和评分快照，便于排查

### 3.4 `video_text_content`

来源：清洗后的简介与字幕

主要写入字段：

- `has_description`
- `has_subtitle`
- `description_text`
- `subtitle_text`
- `combined_text`
- `combined_text_hash`
- `language_code`

说明：

- `combined_text` 是后续 AI 摘要与主题提取的主要输入
- 当字幕缺失时，仍然会保存简介或标题兜底文本

### 3.5 `video_subtitle_segment`

来源：字幕采集器返回的字幕分段

主要写入字段：

- `segment_index`
- `start_seconds`
- `end_seconds`
- `content`

说明：

- 每次写入文本内容时，都会重建该文本记录下的字幕分段
- 同一个 `text_content_id + segment_index` 不会重复

## 4. 流水线改造

阶段 5 已将数据落库正式接入采集流水线：

- [crawl_pipeline_service.py](E:\code\fullstack\spiderbilibili\backend\app\services\crawl_pipeline_service.py)

当前执行顺序是：

1. 重置当前任务已有输出
2. 搜索候选视频
3. 去重并选出目标视频
4. 拉取详情
5. 拉取字幕
6. 计算综合评分
7. 立即持久化到业务表
8. 更新任务摘要和日志

这样做的好处是：

- 单个视频成功时可以尽早入库
- 部分视频失败时，成功结果不会丢
- 任务再次执行时能保持输出一致

## 5. 任务级幂等策略

阶段 5 新增了任务输出重置逻辑。

在同一个任务重新执行时，会先删除该任务已有的：

- `task_video`
- `video_metric_snapshot`
- `video_text_content`
- `video_subtitle_segment`
- `ai_summary`
- `topic_cluster`
- `topic_video_relation`

说明：

- `video` 主表不会删除，因为它是全局视频资产表
- 清理 AI 与主题表是为了给后续阶段重跑留出一致性空间
