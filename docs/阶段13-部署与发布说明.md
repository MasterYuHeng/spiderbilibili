# 阶段13：部署与发布说明

## 1. 目标

阶段 13 的交付目标是让项目具备单机生产部署能力，覆盖以下内容：

- 后端 API 的生产镜像
- Celery Worker 的运行方案
- 前端静态资源构建与托管
- Nginx 统一入口与 API 反向代理
- 可复现的部署、升级、回滚手册

## 2. 部署架构

生产部署使用 `docker-compose.prod.yml` 编排以下服务：

- `postgres`：业务数据库
- `redis`：缓存与 Celery Broker/Backend
- `backend`：FastAPI API 服务，启动时自动执行 Alembic 迁移
- `worker`：Celery Worker，复用后端镜像
- `nginx`：前端静态资源托管 + `/api` 反向代理

统一访问入口由 `nginx` 提供，浏览器只需要访问一个域名或端口。

生产编排显式使用独立的 Compose 项目名 `spiderbilibili-prod`，避免与本地开发环境的 `docker-compose.yml` 共享容器名、网络和数据卷。

## 3. 服务器要求

推荐最低配置如下：

- Linux x86_64 服务器一台
- `2 vCPU / 4 GB RAM / 20 GB SSD`
- Docker `24+`
- Docker Compose v2

如果需要长期运行 AI 分析、批量爬取和 Playwright 兜底链路，推荐配置提升到：

- `4 vCPU / 8 GB RAM`

## 4. 关键文件

- `backend/Dockerfile`：后端与 Worker 共用生产镜像
- `frontend/Dockerfile`：前端构建与 Nginx 运行镜像
- `docker/nginx/default.conf`：Nginx 静态托管与 API 代理配置
- `docker-compose.prod.yml`：生产编排文件
- `docker/.env.production.example`：生产环境变量模板

说明：

- 后端镜像基于官方 `Playwright Python v1.58.0` 镜像，浏览器与系统依赖提前内置，避免在生产构建时重复安装 Chromium 依赖。
- `backend` 和 `worker` 开启了 `init: true` 与 `ipc: host`，用于降低 Chromium 子进程僵死和共享内存不足带来的运行风险。

## 5. 环境变量

部署前先复制生产环境变量模板：

```bash
cp docker/.env.production.example docker/.env.production
```

至少要确认以下变量：

- `WEB_PORT`：对外暴露端口，默认 `80`
- `APP_CORS_ORIGINS`：前端访问域名，例如 `https://your-domain.com`
- `POSTGRES_DB` / `POSTGRES_USER` / `POSTGRES_PASSWORD`
- `OPENAI_API_KEY`
- `OPENAI_BASE_URL` / `OPENAI_MODEL`
- `CRAWLER_ENABLE_PROXY`、`HTTP_PROXY`、`HTTPS_PROXY`：如需代理时配置
- `CELERY_WORKER_CONCURRENCY`：Worker 进程并发数

其余变量未覆盖时会回退到 `docker-compose.prod.yml` 中的默认值。

## 6. 部署命令

### 6.1 校验配置

```bash
docker compose --env-file docker/.env.production -f docker-compose.prod.yml config
```

### 6.2 构建镜像

```bash
docker compose --env-file docker/.env.production -f docker-compose.prod.yml build
```

### 6.3 启动服务

```bash
docker compose --env-file docker/.env.production -f docker-compose.prod.yml up -d
```

### 6.4 查看运行状态

```bash
docker compose --env-file docker/.env.production -f docker-compose.prod.yml ps
docker compose --env-file docker/.env.production -f docker-compose.prod.yml logs -f backend worker nginx
```

### 6.5 验证部署

```bash
curl http://127.0.0.1:80/api/health
```

如果通过域名部署，可直接访问：

- `http://your-domain/`
- `http://your-domain/api/health`

## 7. 升级方式

推荐升级流程如下：

1. 拉取最新代码或切换到目标发布版本。
2. 备份数据库。
3. 重新构建镜像。
4. 使用同一份 `docker/.env.production` 执行滚动重启。

示例命令：

```bash
mkdir -p backups
docker compose --env-file docker/.env.production -f docker-compose.prod.yml exec -T postgres \
  pg_dump -U spider_user spider_bilibili > backups/pre-upgrade.sql

docker compose --env-file docker/.env.production -f docker-compose.prod.yml build
docker compose --env-file docker/.env.production -f docker-compose.prod.yml up -d
```

如果你修改了数据库用户名或数据库名，请把备份命令里的 `spider_user` 和 `spider_bilibili` 替换成你自己的值。

## 8. 回滚方式

推荐回滚策略：

1. 切回上一个已验证通过的代码版本或发布标签。
2. 保留原来的 `docker/.env.production`。
3. 重新构建并启动上一个版本。
4. 如果数据库迁移与旧版本不兼容，再恢复升级前备份。

回滚示例：

```bash
docker compose --env-file docker/.env.production -f docker-compose.prod.yml down
docker compose --env-file docker/.env.production -f docker-compose.prod.yml build
docker compose --env-file docker/.env.production -f docker-compose.prod.yml up -d
```

如需恢复数据库，可在确认容器已启动后执行：

```bash
cat backups/pre-upgrade.sql | docker compose --env-file docker/.env.production -f docker-compose.prod.yml exec -T postgres \
  psql -U spider_user -d spider_bilibili
```

同样请按你的实际数据库账号和库名替换命令中的参数。

## 9. 说明与边界

- 当前方案是单机部署方案，适合中小规模生产环境和内网部署。
- 当前 Nginx 配置负责 HTTP 入口、静态资源缓存和 API 代理，不包含 TLS 证书配置。
- 如果部署到公网，建议在本方案外再接入云负载均衡、证书管理或外层反向代理。
- `backend` 与 `worker` 共享日志卷和原始数据卷，便于排障与追溯。
