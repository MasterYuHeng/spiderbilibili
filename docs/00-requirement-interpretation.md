# 需求解读文档

## 1. 文档定位

本文档是 SpiderBilibili 的项目级需求解读文档，用于把当前代码、现有文档和固定协作文档统一解释为后续协作的主需求边界。
固定约束以 `docs/codex/` 为准；本文档负责承接项目本身的业务目标、范围和当前需求口径。

## 2. 项目目标解读

SpiderBilibili 的目标不是单点爬虫脚本，而是一个面向中文内容研究场景的 Bilibili 任务式采集与分析平台。
项目当前应服务以下主线目标：

- 让用户能够创建和管理 Bilibili 采集任务。
- 支持关键词抓取与热榜抓取两类主要入口。
- 将搜索结果、视频详情、分析结果、报告与验收结果组织为单任务工作流。
- 提供 AI 相关配置与 Bilibili 登录态配置，支持运行期采集与分析。
- 提供本地开发、Windows 一键启动和 Docker 部署能力。

## 3. 当前确认的核心需求范围

基于现有代码、README 和历史专题文档，当前项目级需求范围解释如下。

### 3.1 任务与采集

- 支持创建采集任务，并配置抓取模式、范围和必要参数。
- 支持在任务创建时通过预设选项或自定义输入控制“最近多少天内发布”的发布时间限制。
- 支持任务列表、详情、暂停、恢复、取消、重试、删除、回收站等生命周期管理。
- 支持搜索结果采集、原始数据归档、候选视频处理和结果导出。

### 3.2 分析与展示

- 支持视频结果查看。
- 支持主题分析、UP 主分析、任务报告、上线验收等任务结果视图。
- 支持任务执行进度和搜索口径的可追踪展示。
- 支持用户在主题分析中自定义关键分析指标的权重配置，并在确认后触发重分析。
- 自定义权重必须同步影响主题分析可视化、热点判断、推荐结果和 AI 报告输出。

### 3.3 系统配置

- 支持运行时配置 DeepSeek / OpenAI 兼容模型相关参数。
- 支持配置或导入 Bilibili 登录态。
- 支持健康检查与运行指标暴露。

### 3.4 部署与运行

- 支持 Windows 一键启动本地开发环境。
- 支持手动本地开发模式。
- 支持 Docker 启动基础设施。
- 支持整套容器化部署与监控组件。

### 3.5 已纳入主需求口径的专题增强

“关键词同义补充”已经不再单独作为项目主文档体系维护，而是作为“关键词抓取模式下的搜索召回增强能力”纳入主需求。
该能力的当前稳定口径是：

- 只增强关键词模式的搜索召回。
- 仍然保持“一个任务、一套分析结果、一套报告”。
- 原始关键词仍然是任务主关键词。
- 详细冻结规格和阶段性专题文档已归档到 `docs/archive/keyword-synonym-expansion/`。

## 4. 本轮增量需求口径

本轮新增的项目级需求口径如下：

- 用户可以自定义爆发力、内容深度、社区扩散、完播代理分、综合互动率、主题热度指数等指标的权重。
- 权重修改不是即时覆盖旧结果，而是“编辑 -> 确认 -> 重新开始分析 -> 输出新结果”的任务级分析重算流程。
- 本轮默认沿用现有任务、主题聚类、分析快照和报告快照架构，不重跑抓取链路，重点重算主题分析与报告生成阶段。
- 权重配置必须作为任务级分析配置持久化，后续查看主题分析页和报告页时都应读取同一套当前权重口径。
- 后续 AI 报告生成必须消费新的分析结果和新的权重说明，避免页面与报告口径不一致。

## 5. 核心约束

- 本项目继续遵循 `docs/codex/` 下的固定协作文档。
- 后续需求变更默认先联动更新 `docs/00-04`，再进入代码实施。
- 不将项目降级为一次性玩具脚本或单功能 Demo。
- 未经明确要求，不改变当前任务制、单任务分析链路和现有核心架构。
- 项目级主文档以 `docs/00-05` 为准；专题历史文档只作为补充参考。

## 6. 当前轮次不做的事情

- 不顺手扩展无关业务功能。
- 不改动与本轮需求无关的核心链路。
- 不在没有必要的情况下重写 `docs/codex/` 下的固定协作文档。
- 不把历史专题文档重新提升为项目级主文档。
