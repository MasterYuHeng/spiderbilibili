# 阶段14：运维与可观测性说明

## 1. 阶段目标

阶段 14 的目标不是只保留一个 `/health` 接口，而是把项目补成一套可持续运维的运行体系，覆盖：

- 基础监控：API、Celery Worker、PostgreSQL、Redis、任务失败率、AI 失败率
- 告警机制：关键任务失败即时通知，监控规则可持续触发告警
- 运维脚本：任务重试、数据导出、日志清理、健康检查

## 2. 已交付能力

### 2.1 应用内监控与指标

后端新增了以下运行时能力：

- `GET /api/health`
  - 返回 API / PostgreSQL / Redis / Worker 的组件状态
  - 返回 `active_workers`、`celery_queue_depth`、`task_failure_ratio`、`ai_failure_ratio`
  - 仅当数据库或 Redis 不可用时返回 `503`
- `GET /api/metrics`
  - 暴露 Prometheus 文本指标
  - 包含 HTTP 请求量、HTTP 延迟、Worker 心跳、任务终态计数、AI 分析计数、队列积压、组件健康状态

应用内关键指标包括：

- `spiderbilibili_http_requests_total`
- `spiderbilibili_http_request_duration_seconds`
- `spiderbilibili_component_health_status`
- `spiderbilibili_worker_active`
- `spiderbilibili_worker_heartbeat_age_seconds`
- `spiderbilibili_celery_queue_depth`
- `spiderbilibili_task_terminal_total`
- `spiderbilibili_task_stage_failures_total`
- `spiderbilibili_ai_video_analyses_total`
- `spiderbilibili_runtime_health`

### 2.2 Worker 心跳

Celery Worker 启动后会向 Redis 写入心跳，后端健康检查和指标导出会读取这些心跳数据。

默认参数：

- `MONITORING_REDIS_PREFIX=spiderbilibili:monitoring`
- `MONITORING_WORKER_HEARTBEAT_TTL_SECONDS=150`
- `MONITORING_WORKER_EXPECTED_COUNT=1`

### 2.3 应用内告警

应用内告警通过环境变量驱动，支持：

- 企业微信机器人 Webhook
- 钉钉机器人 Webhook
- SMTP 邮件

当前会在以下场景发出即时告警：

- 任务执行过程抛出异常
- 任务最终状态为 `failed`
- 任务最终状态为 `partial_success`

核心环境变量：

- `ALERTING_ENABLED`
- `ALERTING_WECHAT_WEBHOOK`
- `ALERTING_DINGTALK_WEBHOOK`
- `ALERTING_EMAIL_ENABLED`
- `ALERTING_EMAIL_SMTP_HOST`
- `ALERTING_EMAIL_USERNAME`
- `ALERTING_EMAIL_PASSWORD`
- `ALERTING_EMAIL_FROM`
- `ALERTING_EMAIL_TO`

说明：

- 内建告警做了 Redis 去重，默认 15 分钟窗口内不会重复刷同一类告警
- `APP_PUBLIC_BASE_URL` 建议配置为线上访问域名，便于后续扩展告警跳转链接

### 2.4 Prometheus / Alertmanager / Grafana

生产编排已经新增：

- `prometheus`
- `alertmanager`
- `grafana`
- `redis-exporter`
- `postgres-exporter`

对应文件：

- `docker/prometheus/prometheus.yml`
- `docker/prometheus/alert.rules.yml`
- `docker/alertmanager/alertmanager.yml`
- `docker/grafana/provisioning/datasources/prometheus.yml`
- `docker/grafana/provisioning/dashboards/dashboards.yml`
- `docker/grafana/dashboards/spiderbilibili-overview.json`

当前预置告警规则包括：

- 后端不可抓取
- PostgreSQL 不可用
- Redis 不可用
- Worker 数量低于预期
- Celery 队列积压过高
- 任务失败率过高
- AI 失败率过高

说明：

- Alertmanager 当前采用“安全默认配置”，不会因为未填写邮箱或 Webhook 而启动失败
- 关键业务通知由应用内告警直接负责
- 如果你后续想把 Prometheus 规则告警也直接投递到外部平台，可以继续扩展 `docker/alertmanager/alertmanager.yml`

## 3. 运维脚本

### 3.1 重试任务

```powershell
.\.venv\Scripts\python.exe .\scripts\retry-task.py --task-id <source-task-id>
```

行为：

- 只允许重试 `failed`、`cancelled`、`partial_success` 任务
- 不复用原任务 ID，而是克隆出一个新任务并重新入队
- 新任务 `extra_params.retry_context.retry_of_task_id` 会记录来源任务

### 3.2 直接导出任务数据

```powershell
.\.venv\Scripts\python.exe .\scripts\export-task-data.py --task-id <task-id> --dataset videos --format csv --output .\exports\task-videos.csv
```

支持：

- `--dataset videos|topics|summaries`
- `--format json|csv|excel`
- `--sort-by`
- `--sort-order`
- `--topic`

### 3.3 清理日志

```powershell
.\.venv\Scripts\python.exe .\scripts\cleanup-logs.py --retention-days 7
```

可选：

- `--dry-run`
- `--log-dir`

### 3.4 健康检查

```powershell
.\.venv\Scripts\python.exe .\scripts\check-health.py --base-url http://127.0.0.1:8000/api --include-metrics
```

## 4. 生产部署补充

### 4.1 启动监控栈

```bash
docker compose --env-file docker/.env.production -f docker-compose.prod.yml up -d
```

新增默认端口：

- Prometheus：`9090`
- Alertmanager：`9093`
- Grafana：`3000`

### 4.2 Grafana 登录

默认从环境变量读取：

- `GRAFANA_ADMIN_USER`
- `GRAFANA_ADMIN_PASSWORD`

### 4.3 推荐最小上线配置

建议至少配置：

- `MONITORING_ENABLED=true`
- `MONITORING_WORKER_EXPECTED_COUNT=1`
- `ALERTING_ENABLED=true`
- `ALERTING_WECHAT_WEBHOOK` 或 `ALERTING_DINGTALK_WEBHOOK` 或 SMTP 邮件
