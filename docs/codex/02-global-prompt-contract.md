# Codex 全局提示词契约

## 1. 定位

本文档定义 Codex 在本项目中应遵循的标准任务解释结构。

用户后续即使只使用简单自然语言，Codex 也应尽量将其转换为本契约中的标准任务结构后再执行。

## 2. 通用任务结构

任意任务默认都应解释为以下结构：

- 任务类型
- 本轮唯一主目标
- 固定约束
- 必须参考
- 工作范围
- 交付物
- 验证要求
- 文档同步
- 输出要求

## 3. 固定约束默认值

除非用户明确改变，否则默认固定约束为：

- 以 `docs/00-requirement-interpretation.md` 为准
- 遵循 `docs/codex/` 下固定协作文档
- 不做玩具项目
- 不改变核心需求、核心功能、核心架构
- 除非用户明确要求其他语言，否则文档内容、默认输出、前端用户可见文案默认使用中文
- 技术专有名词、库名、协议名、代码标识、接口字段名、数据库字段名在必要时可保留英文
- 文档优先增量更新
- 每轮任务完成后更新 `docs/04-current-state.md`

## 4. 任务类型

标准任务类型包括：

- 项目初始化
- 需求解释文档
- PRD
- 技术文档
- 实施路线文档
- 旧项目文档治理
- 功能扩展
- 阶段实施
- 环境配置
- 环境检测
- Review
- 技术完成报告
- 需求完成报告
- 项目完成前可运行性检查
- 阶段识别
- Git 初始化
- 开源整理
- 需求变更联动
- 新线程续接

## 5. 任务解释规则

### 5.1 项目初始化

目标：

- 建立项目核心运行文档。

交付物：

- `docs/00-requirement-interpretation.md`
- `docs/01-prd.md`
- `docs/02-tech-stack-and-env.md`
- `docs/03-implementation-roadmap.md`
- `docs/04-current-state.md`

### 5.2 文档类任务

适用类型：

- 需求解释文档
- PRD
- 技术文档
- 实施路线文档
- 需求变更联动

默认规则：

- 如文档已存在，只更新受影响章节。
- 保留未变内容。
- 多份文档联动时，只修改真正受影响的部分。
- 文档标题、正文、说明、使用指南、README 默认使用中文。

### 5.3 旧项目文档治理

适用类型：

- 旧项目文档治理

默认规则：

- 默认适用于已有项目文档混乱、重复、命名不统一、职责不清、缺少主文档的场景。
- 优先暂停无关开发，先做文档整理。
- 先读取当前 docs、核心项目文档和必要代码目录。
- 先判断哪些文档应保留、合并、拆分、归档或废弃。
- 补齐标准主文档，并迁移有效内容。
- 保留有参考价值但不适合作为主文档的历史文档，并归档到合适位置。
- 无必要的英文文档内容、说明文字和前端用户可见文案尽量统一为中文。
- 完成后更新 `docs/04-current-state.md`。

### 5.4 功能扩展

适用类型：

- 功能扩展

默认规则：

- 默认适用于已有项目新增模块、新增子系统、新增业务流程、扩展现有能力。
- 先判断扩展目标、影响范围、依赖关系、实现边界。
- 先联动更新受影响文档，再进入实现。
- 扩展时不得破坏现有核心功能、核心架构和已有稳定链路。
- 需要明确哪些内容是新增能力，哪些内容是兼容性改动，哪些内容禁止改动。
- 如扩展涉及前端，前端用户可见文案默认使用中文。

### 5.5 实施类任务

适用类型：

- 阶段实施
- 环境配置
- 环境检测

默认规则：

- 直接落地到代码和配置文件。
- 不把全局 Review 和总报告混入当前轮次。
- 只读取本轮必要文件和目录。
- 若实现与文档存在小范围偏差，可同步修正文档。
- 前端用户可见文案，包括标题、按钮、表单标签、占位符、提示信息、空状态、错误信息，默认使用中文。

### 5.6 检查类任务

适用类型：

- Review
- 技术完成报告
- 需求完成报告
- 项目完成前可运行性检查
- 阶段识别

默认规则：

- 可扩大读取范围。
- 结论必须基于当前代码实际状态。
- 不仅看文档，也要看实际文件和验证结果。
- 如任务类型为“项目完成前可运行性检查”，应明确判断项目是否基本完成、是否可以成功运行、当前阻塞点是什么、是否适合进入 Git 备份阶段。

### 5.7 管理类任务

适用类型：

- Git 初始化
- 开源整理

默认规则：

- 在项目基本可用后执行。
- 先检查现状，再执行整理或初始化。

## 6. 工作范围解释规则

Codex 应从用户自然语言中优先提取：

- 允许读取的路径
- 允许修改的路径
- 明确不能改的范围

如果用户没有说，但可从上下文稳定推断，则直接推断。

如果无法稳定推断，且误改风险较高，则追问用户。

## 7. 验证规则

默认验证顺序为：

1. 配置检查
2. 依赖检查
3. 构建检查
4. 测试检查
5. 启动检查

并非每轮都要执行全部验证。

Codex 应根据任务类型做最小必要验证。

如果任务类型为“项目完成前可运行性检查”，默认应尽量覆盖：

1. 配置检查
2. 依赖检查
3. 构建检查
4. 核心测试检查
5. 启动检查
6. 核心功能链路检查

如果无法验证，应明确说明：

- 哪一步无法验证
- 阻塞原因
- 当前已确认的范围

## 8. 文档同步规则

每轮任务完成后，至少同步：

- `docs/04-current-state.md`

如涉及关键技术决策变化，再同步：

- `docs/05-decision-log.md`

如涉及需求变化，再联动：

- `docs/00-requirement-interpretation.md`
- `docs/01-prd.md`
- `docs/02-tech-stack-and-env.md`
- `docs/03-implementation-roadmap.md`

## 9. 新线程规则

用户在新线程中即使只说一句简短自然语言，Codex 也应默认先恢复以下上下文：

- 固定协作文档
- 需求解释文档
- 实施路线文档
- 当前状态文档
- 本轮直接相关目录

然后再执行任务。

## 10. 输出契约

每轮输出默认包含：

- 本轮完成了什么
- 验证了什么
- 当前剩余风险
- 下一步建议

每轮输出默认语言：

- 中文优先
- 必要英文仅限技术专有内容

不应默认输出：

- 大段重复背景
- 与本轮无关的全量总结
- 无必要的重复文档重写
