# Codex 全局工作流

## 1. 定位

本文档是项目内固定协作文档。

作用：

- 约束 Codex 在本项目中的标准工作方式。
- 统一项目从启动到交付的推进流程。
- 降低重复解释成本。
- 降低新线程续接成本。

非作用：

- 不替代项目需求文档。
- 不替代 PRD。
- 不替代当前状态文档。

## 2. 固定原则

- 本文档及同目录下的固定协作文档默认长期有效。
- 除非用户明确要求，否则 Codex 不应重写这些固定协作文档。
- 后续所有任务默认遵循这些固定协作文档。
- 用户后续可用简单自然语言下达任务。
- Codex 应先按本套文档解释任务，再执行。
- 除非用户明确要求其他语言，否则文档内容、默认输出、前端用户可见文案默认使用中文。
- 技术专有名词、库名、协议名、代码标识、接口字段名、数据库字段名在必要时可保留英文。

## 3. 固定协作文档

固定协作文档目录：

- `docs/codex/01-global-workflow.md`
- `docs/codex/02-global-prompt-contract.md`
- `docs/codex/03-natural-language-routing.md`
- `docs/codex/04-usage-guide.md`

这些文档的性质：

- 属于项目级固定规则层。
- 可直接复制到新项目中复用。
- 主要用于约束 Codex 的工作逻辑。
- 旧项目文档治理和项目功能扩展的分支规则，已吸收到本套固定协作文档中，不再独立维护。

## 4. 项目运行期文档

Codex 在项目推进中应自动创建并维护以下项目运行期文档：

- `docs/00-requirement-interpretation.md`
- `docs/01-prd.md`
- `docs/02-tech-stack-and-env.md`
- `docs/03-implementation-roadmap.md`
- `docs/04-current-state.md`
- `docs/05-decision-log.md`
- `docs/90-tech-completion-report.md`
- `docs/91-requirement-completion-report.md`

其中最关键的是：

- `docs/04-current-state.md`

要求：

- 每轮任务完成后更新。
- 写明当前阶段、已完成项、当前阻塞、下一步建议。
- 新线程续接时优先读取。

## 5. 标准执行顺序

正常项目按以下顺序推进：

1. 建立需求解释文档。
2. 建立 PRD。
3. 建立技术栈与环境文档。
4. 建立实施路线文档。
5. 建立当前状态文档。
6. 按阶段实施。
7. 完成环境配置与环境检测。
8. 做阶段性 Review 与修复。
9. 输出技术完成报告与需求完成报告。
10. 做一次项目完成前可运行性检查，确认项目是否基本完成、核心链路是否可用、项目是否可以成功运行。
11. 在项目基本可用后初始化 Git 并完成备份。
12. 在需要开源时整理为开源形态。

如果是在旧项目上先整理混乱文档，标准顺序为：

1. 暂停无关功能开发。
2. 梳理当前 docs、核心项目文档和必要代码目录。
3. 判断哪些文档应保留、合并、拆分、归档或废弃。
4. 补齐标准主文档。
5. 迁移有效内容，清理重复内容。
6. 无必要的英文文档内容和前端用户可见文案尽量统一为中文。
7. 更新当前状态文档。
8. 再恢复正常开发推进。

如果是在已有项目上做功能扩展，标准顺序为：

1. 明确扩展目标和扩展边界。
2. 判断对现有需求、PRD、技术方案、实施路线和当前状态的影响。
3. 先更新受影响文档。
4. 再按阶段实施扩展功能。
5. 完成必要验证。
6. 更新当前状态文档。

项目完成前可运行性检查至少应覆盖：

1. 当前项目是否已达到“基本完成”标准。
2. 环境配置是否齐全。
3. 依赖是否完整。
4. 构建是否成功。
5. 测试是否通过或至少核心测试是否通过。
6. 启动流程是否成功。
7. 核心功能链路是否可正常运行。
8. 当前是否仍存在阻止 Git 备份或后续交付的关键阻塞。

## 6. 每轮任务规则

每轮只处理一个主目标。

每轮都应明确：

- 本轮目标
- 必须参考的文档
- 必须参考的目录
- 允许修改的范围
- 非本轮范围
- 交付物
- 验证要求

每轮结束都应输出：

- 修改内容
- 验证结果
- 剩余风险
- 下一步建议

并同步更新：

- `docs/04-current-state.md`

## 7. 上下文控制规则

- 固定约束不在对话里重复长篇复述，默认引用本目录文档。
- 普通开发轮次只读取相关文档和相关目录。
- 不在普通开发轮次全量扫描仓库。
- 文档优先增量更新，不重写整份文档。
- 只有在 Review、报告、开源整理、阶段识别时扩大扫描范围。

## 8. 新线程续接规则

新线程开始时，默认先读取：

- `docs/codex/01-global-workflow.md`
- `docs/codex/02-global-prompt-contract.md`
- `docs/codex/03-natural-language-routing.md`
- `docs/00-requirement-interpretation.md`
- `docs/03-implementation-roadmap.md`
- `docs/04-current-state.md`

然后只补充读取本轮直接相关目录。

新线程中：

- 不重新复述全项目背景。
- 不重新解释固定规则。
- 直接基于当前状态继续推进。

## 9. 需求变更规则

如果用户提出新增或变更需求，执行顺序应为：

1. 更新 `docs/00-requirement-interpretation.md`
2. 更新 `docs/01-prd.md`
3. 更新 `docs/02-tech-stack-and-env.md`
4. 更新 `docs/03-implementation-roadmap.md`
5. 更新 `docs/04-current-state.md`
6. 再进入代码实施

除非用户明确要求，否则不应跳过文档联动直接改代码。

如果用户是在旧项目上先治理文档，默认也按同样原则处理：

- 先完成文档治理
- 再恢复功能开发

如果用户是在已有项目上扩展功能，默认也按同样原则处理：

- 先判断扩展影响范围
- 再联动更新文档
- 最后进入实施

## 10. Codex 默认行为

在本项目中，Codex 默认应这样工作：

- 将用户简单自然语言解释为标准任务包。
- 尽量基于现有文档推断缺省信息。
- 只有在高风险缺口存在时才追问用户。
- 优先直接落地文件，而不是只给建议。
- 发现局部规划不合理时，可小幅修正文档，但不得改变核心需求、核心功能、核心架构。
- 生成文档、README、说明文字、前端按钮文案、表单标签、占位符、提示信息、空状态、错误信息时，默认优先使用中文。

## 11. 最短执行版

如果只看最短规则，按下面执行：

1. 固定规则看 `docs/codex/`
2. 项目状态看 `docs/04-current-state.md`
3. 每轮只做一个主目标
4. 只读必要文件
5. 直接改文件
6. 做必要验证
7. 更新当前状态文档
