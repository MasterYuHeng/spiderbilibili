# Codex 使用说明

## 1. 这四份文档怎么用

本项目中：

- `docs/codex/01-global-workflow.md`
- `docs/codex/02-global-prompt-contract.md`
- `docs/codex/03-natural-language-routing.md`
- `docs/codex/04-usage-guide.md`

共同构成固定协作文档。

作用：

- 约束 Codex 的工作方式
- 让你后续尽量只用简单自然语言交流
- 减少重复解释
- 降低新线程切换成本

固定协作文档结构只保留 `01` 到 `04`。

旧项目文档治理和项目功能扩展这两类分支场景，已经合并到 `01` 到 `04` 中。

## 2. 这四份文档中哪三份是固定核心文档

固定核心文档是：

- `docs/codex/01-global-workflow.md`
- `docs/codex/02-global-prompt-contract.md`
- `docs/codex/03-natural-language-routing.md`

本文件是使用说明文档。

## 3. 在一个新项目中如何使用

开始一个新项目时：

1. 把 `docs/codex/` 目录复制到新项目中。
2. 开始与 Codex 交互。
3. 第一轮先明确：本项目遵循 `docs/codex/` 下固定协作文档。
4. 让 Codex 先建立项目运行期文档。
5. 后续直接用简单自然语言推进。

## 4. 是否需要向 Codex 强调这些是全局固定文档

需要。

但不需要每轮都长篇强调。

最推荐的强调方式：

- 在每个新项目的第一轮强调一次。
- 在每个新线程的第一轮简短强调一次。
- 平时普通轮次不再重复强调。
- 如你希望整个项目默认中文，也在第一轮一并强调一次。

## 5. 第一次进入项目时的推荐说法

你可以直接对 Codex 说：

```md
本项目以 `docs/codex/` 下文档作为固定协作文档。请默认遵循这些文档，不要重写这些固定文档，除非我明确要求。后续请优先基于这些文档和项目运行期文档理解我的简单自然语言任务，并直接执行。除非我明确要求其他语言，否则文档、默认输出和前端用户可见文案请默认使用中文，必要的技术专有名词可保留英文。
```

这句话的作用已经足够。

但如果这是你第一次创建这个项目，仅强调固定协作文档还不够，你还需要把项目需求包装成首轮启动提示词。

### 5.1 第一次创建项目时，你至少要提供什么

第一次创建项目时，建议你至少提供以下内容：

- 项目名称
- 项目要解决的核心问题
- 核心用户或核心角色
- 核心功能
- 明确不能改变的核心需求
- 明确不能改变的核心架构边界
- 非功能要求
- 如有已知技术限制，也一并说明

如果你一时说不全，也至少要说清下面四项：

- 这个项目是做什么的
- 最核心的功能是什么
- 哪些内容绝对不能被改成简化版
- 对性能、稳定性、扩展性有没有明确要求

### 5.2 第一次创建项目时的推荐提示词

你可以直接说：

```md
本项目以 `docs/codex/` 下文档作为固定协作文档。请默认遵循这些文档，不要重写这些固定文档，除非我明确要求。

现在这是一个新项目，请先根据我的需求建立项目运行期文档，并把项目基线定下来。

我的需求如下：
- 项目名称：[填写]
- 项目目标：[填写]
- 核心用户或角色：[填写]
- 核心功能：[填写]
- 不允许改变的核心需求：[填写]
- 不允许改变的核心架构边界：[填写]
- 非功能要求：[填写]
- 其他限制或补充：[填写，没有就写“无”]

执行要求：
- 先建立并输出项目运行期文档
- 不做玩具项目，不做临时性方案
- 除非我明确要求其他语言，否则文档、默认输出和前端用户可见文案默认使用中文，必要技术专有名词可保留英文
- 如果我的需求有歧义，可以做合理工程化补充，但要明确标注为“补充假设”
- 先以文档建立和项目基线固化为主，再给我下一步建议
```

### 5.3 如果你只想用更短的版本

你也可以直接说：

```md
本项目以 `docs/codex/` 下文档作为固定协作文档。现在这是一个新项目，请先根据我的需求建立项目运行期文档并定下项目基线。

我的需求：
- 这个项目要做：[填写]
- 核心功能：[填写]
- 绝对不能改变的内容：[填写]
- 非功能要求：[填写]

除非我明确要求其他语言，否则文档、默认输出和前端用户可见文案默认使用中文，必要技术专有名词可保留英文。先把需求解释、PRD、技术栈与环境、实施路线、当前状态这些核心文档建立起来。
```

### 5.4 如何把你的自然语言包装得更适合首轮输入

最推荐的写法不是一整段随意描述，而是按下面顺序写：

1. 先声明 `docs/codex/` 是固定协作文档。
2. 再声明这是新项目。
3. 再写项目目标。
4. 再写核心功能。
5. 再写不可变边界。
6. 再写非功能要求。
7. 最后要求 Codex 先建立项目运行期文档。

只要你按这个顺序组织，首轮提示词通常就已经足够稳。

## 6. 新线程时的推荐说法

你可以直接说：

```md
这是新线程。本项目继续遵循 `docs/codex/` 下固定协作文档。请先读取固定协作文档、`docs/00-requirement-interpretation.md`、`docs/03-implementation-roadmap.md`、`docs/04-current-state.md`，再继续处理我下面的任务。
```

然后下一句直接说本轮任务即可。

## 7. 后续日常怎么用

后续你可以尽量只使用简单自然语言。

Codex 应自己完成：

- 识别任务类型
- 识别主目标
- 识别文档和目录范围
- 推断默认约束
- 执行必要验证
- 同步更新当前状态文档

## 8. 你不需要每轮重复说的话

后续通常不需要重复说：

- 不要做玩具项目
- 不要改核心需求
- 参考之前流程
- 参考之前模板
- 按固定协作文档来

因为这些都已经固化在 `docs/codex/` 中。

## 9. 你仍然需要在自然语言里说清楚的内容

你每轮至少要说清楚：

- 这轮要做什么
- 大概做哪个阶段或哪个模块
- 是否有限制范围

如果你要临时改语言，也要说清楚：

- 哪些部分改成英文或双语
- 哪些部分仍保持中文

例如：

- 继续做第 2 阶段的认证模块，只改后端。
- 先补 PRD 里的异常流程，不要动代码。
- review 一下支付模块并直接修复。
- 检查当前环境是否满足后续开发。

## 10. 常见分支场景

### 10.1 旧项目文档治理

适用场景：

- 项目已有较多历史文档
- 文档结构混乱
- 文档内容重复
- 文档命名不统一
- 需要在不直接开始写代码前，先完成文档治理

标准自然语言模板：

```md
本项目继续遵循 `docs/codex/` 下固定协作文档。现在先暂停功能开发，优先做一次旧项目文档治理。

请先读取当前项目中的 docs 和核心项目文档，按固定协作文档的规则完成一次文档梳理、重组和规范化。

本轮要求：
- 先梳理现有文档
- 判断哪些文档应保留、合并、拆分、归档或废弃
- 建立或补齐标准主文档
- 迁移有效内容，清理重复内容
- 无必要的英文文档内容、说明文字和前端用户可见文案尽量规范为中文
- 保留有参考价值但不适合作为主文档的历史文档，并归档到合适位置
- 从本轮开始，后续所有文档都遵循 `docs/codex/` 下固定协作文档

执行要求：
- 本轮以文档治理为主，不要扩展到无关代码开发
- 如文档整理需要少量代码核对，可以读取必要代码，但不要顺手做无关实现
- 直接修改和整理文件，不要只给建议
- 完成后更新 `docs/04-current-state.md`

输出要求：
- 说明新的文档结构
- 说明哪些旧文档被保留、合并、归档或废弃
- 说明补齐了哪些标准主文档
- 说明当前项目下一步应该从哪里继续推进
```

精简版自然语言模板：

```md
本项目继续遵循 `docs/codex/` 下固定协作文档。先暂停开发，优先整理当前所有 docs。请按固定规则完成文档治理、重组和归档，补齐标准主文档，并尽量把无必要的英文文档内容和前端用户可见文案规范为中文，让后续所有文档都按这套规则维护。完成后更新 `docs/04-current-state.md`。
```

强约束版自然语言模板：

```md
本项目继续遵循 `docs/codex/` 下固定协作文档。现在先只做旧项目文档治理，不做无关代码开发。

请读取当前 docs 和必要的核心代码目录，完成一次严格的文档重组：
- 统一文档结构
- 统一文档职责
- 消除重复内容
- 无必要的英文文档内容和前端用户可见文案尽量统一为中文
- 归档过程性和历史性文档
- 补齐标准主文档
- 更新 `docs/04-current-state.md`

除非文档治理必须依赖代码核对，否则不要修改业务代码。
```

治理完成后建议接话：

```md
按整理后的文档体系继续推进，直接开始 [阶段/模块/功能]。
```

或：

```md
读取整理后的文档和当前状态，告诉我现在项目处于什么阶段，下一步先做什么。
```

### 10.2 功能扩展

适用场景：

- 给已有项目增加一个新功能
- 给已有项目增加一个新模块
- 给已有项目增加一个新子系统
- 在现有系统上补充新的业务流程
- 在保持现有主链路稳定的前提下扩展能力

标准自然语言模板：

```md
本项目继续遵循 `docs/codex/` 下固定协作文档。现在我需要在已有项目上扩展一个新功能。

扩展目标：
- 新功能或新模块名称：[填写]
- 这个扩展要解决的问题：[填写]
- 预期服务的角色或用户：[填写]
- 核心扩展能力：[填写]
- 明确不能破坏的现有功能或链路：[填写]
- 明确不能改变的核心架构边界：[填写]
- 其他限制或补充：[填写，没有就写“无”]

执行要求：
- 先读取固定协作文档、项目运行期文档和本扩展直接相关目录
- 先判断这次功能扩展对需求、PRD、技术方案、实施路线、当前状态的影响
- 先联动更新受影响文档
- 再按现有架构完成功能扩展实施
- 不要破坏现有核心功能、核心架构和稳定链路
- 除非我明确要求其他语言，否则文档、默认输出和前端用户可见文案默认使用中文，必要技术专有名词可保留英文
- 完成后更新 `docs/04-current-state.md`

输出要求：
- 说明扩展影响范围
- 说明更新了哪些文档
- 说明实现了哪些扩展内容
- 说明验证结果
- 说明剩余风险和下一步建议
```

文档优先版模板：

```md
本项目继续遵循 `docs/codex/` 下固定协作文档。现在我要在已有项目上扩展 [新功能/新模块]，但这轮先不要急着实现代码。

请先读取固定协作文档、项目运行期文档和相关目录，判断这次功能扩展的影响范围，并先联动更新需求、PRD、技术方案、实施路线和当前状态文档。等文档整理好后，再告诉我下一步应该怎么实施。
```

直接扩展实施版模板：

```md
本项目继续遵循 `docs/codex/` 下固定协作文档。现在请在已有项目上直接扩展 [新功能/新模块]。

要求：
- 先判断影响范围
- 先更新受影响文档
- 再直接完成本轮可落地的扩展实现
- 不要破坏现有核心功能和核心架构
- 只读取本轮必要文件和目录
- 完成后做必要验证并更新 `docs/04-current-state.md`
```

限定范围扩展版模板：

```md
本项目继续遵循 `docs/codex/` 下固定协作文档。现在在已有项目上扩展 [新功能/新模块]。

限制：
- 只改 [目录/模块]
- 不要动 [目录/模块]
- 先更新受影响文档
- 再完成扩展实现
- 不要破坏现有稳定链路
```

新线程扩展版模板：

```md
这是新线程。本项目继续遵循 `docs/codex/` 下固定协作文档。请先读取固定协作文档、`docs/00-requirement-interpretation.md`、`docs/01-prd.md`、`docs/03-implementation-roadmap.md`、`docs/04-current-state.md` 和本扩展相关目录，再继续扩展 [新功能/新模块]。
```

扩展完成后建议检查：

- 是否破坏了现有核心功能
- 是否偏离了核心架构
- 是否同步更新了需求、PRD、技术文档、实施路线、当前状态
- 是否补齐了必要的验证
- 是否还有兼容性风险

## 11. 推荐的简单自然语言模板

你后续只需要在下面模板里替换少量内容即可。

### 11.1 启动项目

```md
本项目遵循 `docs/codex/` 下固定协作文档。除非我明确要求其他语言，否则文档、默认输出和前端用户可见文案默认使用中文。根据我的需求，先把项目运行期文档建立起来。
```

### 11.2 生成或更新文档

```md
按固定协作文档，帮我生成或更新 [文档名/文档主题]，先只改文档，不要动代码。
```

### 11.3 继续做某阶段或某模块

```md
按当前状态继续推进，直接完成 [阶段/模块/功能]。
```

### 11.4 限定范围实施

```md
按当前文档继续做 [模块/功能]，只改 [目录/范围]，不要动 [目录/范围]。
```

### 11.5 功能扩展

```md
在现有项目上扩展 [新功能/新模块/新子系统]。请先判断影响范围，联动更新受影响文档，再按当前架构完成扩展实施，不要破坏现有核心功能和核心架构。
```

### 11.6 环境配置

```md
按当前文档完成环境配置，并检查环境是否满足后续开发。
```

### 11.7 Review

```md
review 一下 [模块/目录]，直接修复本轮能确定的问题，并同步必要文档。
```

### 11.8 阶段识别

```md
读取当前项目和核心文档，告诉我现在做到哪个阶段，还缺什么，下一步该做什么。
```

### 11.9 技术完成报告

```md
根据当前代码和文档，输出技术完成报告。
```

### 11.10 需求完成报告

```md
根据当前代码和文档，输出需求完成报告。
```

### 11.11 项目完成前可运行性检查

```md
在进入 Git 备份前，先检查当前项目是否已经基本完成，并确认项目现在是否可以成功运行。请基于当前代码、核心文档、环境配置、构建、测试、启动和核心功能链路给出结论，并说明当前阻塞点。
```

### 11.12 Git 初始化

```md
当前项目基本可用，初始化本地 Git 仓库并完成一次备份。
```

### 11.13 开源整理

```md
把当前项目整理成适合开源发布的形式，保留完整功能，补齐 README、环境配置教程和 Docker 方案。
```

### 11.14 新线程续接

```md
这是新线程。先按固定协作文档和当前状态恢复上下文，然后继续做 [阶段/模块/功能]。
```

### 11.15 需求变更

```md
现在新增或变更一个需求：[填写]。先联动更新项目文档，再告诉我后续实现应该怎么推进。
```

## 12. 最推荐的实际交互方式

最推荐的实际做法是：

1. 首轮强调一次固定协作文档有效。
2. 让 Codex 建立项目运行期文档。
3. 后续每轮用一句到两句自然语言下达任务。
4. 新线程先恢复固定文档和当前状态。
5. 项目接近完成时，先做一次项目完成前可运行性检查，再进入 Git 备份。
6. 只在需求明显变化时，先更新文档再继续实施。

## 13. 最后提醒

如果你发现 Codex 在某轮偏离了固定规则，直接补一句：

```md
请按 `docs/codex/` 下固定协作文档重新解释并执行本轮任务。
```

通常这一句就够了。
