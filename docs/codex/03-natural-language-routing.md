# 自然语言路由规则

## 1. 定位

本文档用于把用户的简单自然语言，路由为标准任务类型与标准执行结构。

目标：

- 让用户尽量少写模板。
- 让 Codex 尽量稳定执行。
- 降低 token 消耗。

## 2. 总规则

用户自然语言默认按以下顺序解释：

1. 识别任务类型
2. 识别唯一主目标
3. 识别必须参考的文档或目录
4. 识别允许修改范围
5. 识别非本轮范围
6. 识别交付物
7. 识别验证要求

如果用户没有明确指定语言：

- 默认将文档内容、默认输出、前端用户可见文案解释为中文优先
- 仅在技术专有内容必须保留时使用英文

如果信息不全：

- 优先从项目文档中补足
- 其次从当前状态中补足
- 最后才追问用户

## 3. 自然语言到任务类型

### 3.1 项目启动类

典型表达：

- 从零开始做这个项目
- 先把项目文档建起来
- 先建立需求和 PRD

路由到：

- 项目初始化

### 3.2 文档类

典型表达：

- 生成需求解释文档
- 补一下 PRD
- 输出技术栈文档
- 整理实施路线
- 先改文档，不急着写代码

路由到：

- 需求解释文档
- PRD
- 技术文档
- 实施路线文档
- 需求变更联动

### 3.3 旧项目文档治理类

典型表达：

- 先整理当前项目文档
- 现在先不要写代码，先把 docs 治理一下
- 这个项目的 docs 很乱，先重组文档
- 先梳理旧文档并补齐标准主文档
- 先做旧项目文档治理

路由到：

- 旧项目文档治理

### 3.4 功能扩展类

典型表达：

- 给项目加一个新功能
- 在现有项目上扩展后台管理
- 增加支付模块
- 给当前系统补一个消息通知中心
- 在不改变核心架构的前提下扩展 [功能]

路由到：

- 功能扩展

### 3.5 实施类

典型表达：

- 直接做第 X 阶段
- 把登录模块做完
- 把后端认证这块落地
- 按文档继续推进

路由到：

- 阶段实施

### 3.6 环境类

典型表达：

- 把环境配好
- 检查环境有没有问题
- 跑一下构建和测试

路由到：

- 环境配置
- 环境检测

### 3.7 检查类

典型表达：

- review 一下当前代码
- 看看目前做到哪一步
- 给我一个完成情况报告
- 检查当前项目是否已经基本完成
- 检查项目现在能不能成功运行
- 在 Git 之前做一次运行检查

路由到：

- Review
- 阶段识别
- 技术完成报告
- 需求完成报告
- 项目完成前可运行性检查

### 3.8 管理类

典型表达：

- 初始化 git
- 备份一下当前项目
- 整理成适合开源的形式

路由到：

- Git 初始化
- 开源整理

### 3.9 新线程续接类

典型表达：

- 这是新线程，接着上次继续
- 按当前状态继续做
- 接着做支付模块

路由到：

- 新线程续接

## 4. 自然语言字段提取规则

### 4.1 主目标

主目标必须唯一。

如果用户一次说了多个动作，Codex 默认按以下规则处理：

- 优先选最明确的主任务先执行
- 其余任务记为下一步建议

如果多个动作互相依赖，则先执行依赖更前的动作。

### 4.2 参考范围

从用户自然语言中提取：

- 阶段名
- 模块名
- 功能名
- 文档名
- 目录名

如果用户说：

- 按文档继续

默认读取：

- `docs/03-implementation-roadmap.md`
- `docs/04-current-state.md`

如果用户说：

- 先整理当前项目文档
- 先做旧项目文档治理

默认读取：

- `docs/00-requirement-interpretation.md`
- `docs/01-prd.md`
- `docs/02-tech-stack-and-env.md`
- `docs/03-implementation-roadmap.md`
- `docs/04-current-state.md`
- 当前 `docs/` 目录
- 必要的核心代码目录

如果用户说：

- 给项目扩展一个新功能
- 在现有系统上增加一个模块

默认读取：

- `docs/00-requirement-interpretation.md`
- `docs/01-prd.md`
- `docs/02-tech-stack-and-env.md`
- `docs/03-implementation-roadmap.md`
- `docs/04-current-state.md`
- 该扩展功能直接相关目录

如果用户说：

- 检查当前项目是否已经基本完成
- 检查项目现在能不能成功运行
- 在 Git 之前做一次运行检查

默认读取：

- `docs/00-requirement-interpretation.md`
- `docs/01-prd.md`
- `docs/02-tech-stack-and-env.md`
- `docs/03-implementation-roadmap.md`
- `docs/04-current-state.md`
- 核心代码目录
- 关键配置目录

### 4.3 修改范围

如果用户说：

- 只改后端
- 不要动前端
- 只改文档

应分别解释为：

- 限制允许修改目录
- 设定非本轮范围
- 仅执行文档类任务

### 4.4 验证要求

如果用户说：

- 做完检查一下
- 跑一下测试
- 确认能启动

应分别解释为：

- 最小必要验证
- 测试验证
- 启动验证

如果用户没有说，Codex 也应做与任务匹配的最小必要验证。

### 4.5 语言

如果用户说：

- 用中文
- 尽量中文
- 文档和前端都用中文

应解释为：

- 文档正文默认中文
- 默认输出默认中文
- 前端用户可见文案默认中文

如果用户没有明确说：

- 仍按固定协作文档默认采用中文优先

## 5. 自然语言默认补全规则

当用户自然语言较短时，Codex 默认补全以下内容：

- 固定约束来自 `docs/codex/` 与 `docs/00-requirement-interpretation.md`
- 项目当前阶段来自 `docs/04-current-state.md`
- 实施顺序来自 `docs/03-implementation-roadmap.md`
- 风险较低的局部文档同步默认允许执行
- 文档内容、默认输出、前端用户可见文案默认采用中文

## 6. 推荐的用户自然语言形式

用户最推荐这样说：

- 先做旧项目文档治理，整理当前 docs 并补齐标准主文档
- 继续做第 2 阶段的用户认证模块
- 在现有项目上扩展一个消息通知模块，先更新文档再实施
- 先补 PRD 里的权限和异常流程
- 检查当前环境是否满足后续开发
- 检查当前项目是否已经基本完成，并确认现在能不能成功运行
- review 一下支付模块并直接修复
- 读取当前状态，告诉我现在做到哪一步
- 这是新线程，接着当前状态继续做后台管理模块

## 7. 不推荐的用户自然语言形式

不推荐这样说：

- 你把整个项目从头到尾都处理一下
- 你先看看然后自己决定做什么
- 你把文档、代码、环境、review、报告一次全做完
- 你应该知道我们之前说过什么

## 8. 高风险追问条件

只有出现以下情况时，Codex 才应优先追问用户：

- 核心需求存在冲突
- 两个模块都有高概率被误改
- 用户要求与固定文档明显冲突
- 关键路径缺少必要输入

除此之外，应优先直接执行。

## 9. 最短路由版

用户简单自然语言默认按下面理解：

- 先判断任务类型
- 再锁定唯一主目标
- 再读固定文档和当前状态
- 再读本轮相关目录
- 再直接执行
