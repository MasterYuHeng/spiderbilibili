# B站关键词视频采集与分析平台 PRD + 技术方案

## 1. 项目概述

### 1.1 项目名称
B站关键词视频采集与分析平台

### 1.2 项目目标
构建一个前后端联动的系统，用户输入关键词后，系统能够以合规、可控、可扩展的方式采集 B站 中与该关键词相关的视频内容，并基于 AI 对视频文本内容进行总结、主题提取、主题分类和热度统计分析，最终通过简洁前端进行展示。

### 1.3 项目价值
- 帮助用户快速了解某个关键词在 B站 的内容分布和热点方向
- 将原始视频信息转化为可阅读、可归纳、可量化的数据结果
- 为选题研究、竞品分析、舆情观察、内容策划提供基础数据能力

## 2. 产品定位

### 2.1 目标用户
- 内容运营人员
- 视频选题研究人员
- 自媒体从业者
- 市场研究或舆情分析人员
- 对 B站 某类内容生态感兴趣的个人用户

### 2.2 核心问题
用户希望搜索某一关键词在 B站 中出现的视频，但仅依赖人工搜索存在以下问题：
- 结果多，筛选成本高
- 无法批量提取标题、简介、统计数据
- 无法快速总结大量视频内容
- 无法按主题进行归类分析
- 无法对不同主题的热度进行定量统计

## 3. 需求范围

### 3.1 核心功能
1. 用户输入关键词并发起采集任务
2. 系统采集与关键词相关的视频列表
3. 系统获取视频标题、视频文案、视频基础数据
4. 系统使用 AI 对视频文案进行总结和主题提取
5. 系统对全部视频主题进行聚类/分类和统计
6. 系统按主题汇总热视频数据并展示分析结果
7. 前端以列表和图表方式展示任务结果

### 3.2 本期不做
- 大规模账号体系运营
- 评论区深度语义分析
- 视频音频自动下载与完整 ASR 转写
- 多平台联动采集
- 实时流式监控

## 4. 关键定义与业务假设

### 4.1 “视频文案”定义
由于 B站 视频并非都公开完整脚本，第一版建议将“视频文案”定义为以下字段的有序组合：
1. 视频简介 / 描述文案
2. 搜索结果中的摘要信息
3. 可公开获取的字幕内容

补充说明：
- 若视频存在公开字幕，则优先纳入分析
- 若无字幕，先使用简介与摘要做总结
- 第一版不默认做音频抽取和 ASR，避免复杂度和合规风险显著上升
- 后续可增加“用户授权后对公开视频做本地 ASR”能力

### 4.2 “相关性 + 热度”定义
采集结果排序不只看搜索命中，还要综合热视频指标。建议使用综合评分：

`综合分 = 相关性分 * 0.6 + 热度分 * 0.4`

其中：
- 相关性分：来自搜索排序位置、标题命中、简介命中、标签命中
- 热度分：由播放、点赞、投币、收藏、弹幕、评论、分享等指标归一化后加权得到

权重可在后台配置，第一版先使用固定权重。

## 5. 详细功能 PRD

### 5.1 关键词采集任务

#### 功能描述
用户在前端输入关键词，配置采集数量、排序策略、是否启用代理、采集间隔等参数，提交后由后端创建任务。

#### 输入项
- 关键词
- 最大采集视频数
- 采集页数上限
- 排序策略
- 请求间隔范围
- 是否启用代理池

#### 输出项
- 任务 ID
- 任务状态
- 任务进度
- 错误信息

### 5.2 视频采集

#### 采集字段
- 关键词
- 视频 ID（`bvid` / `aid`）
- 标题
- 视频链接
- UP 主名称
- 发布时间
- 视频简介
- 标签（若可获取）
- 时长
- 播放量
- 点赞量
- 投币量
- 收藏量
- 分享量
- 弹幕量
- 评论量
- 搜索排名位置
- 原始抓取时间

#### 采集策略
- 优先使用搜索结果页获取候选视频
- 对候选视频详情页进一步补充数据
- 对可获取字幕的视频补充字幕文本
- 对重复视频按 `bvid` 去重

### 5.3 AI 总结与主题提取

#### 单视频处理输出
- 视频摘要
- 主题词
- 一级主题
- 情绪/表达风格（可选）
- 信息密度标签（可选）

#### 处理逻辑
- 对过长文本先做清洗和截断
- 对每个视频生成 100-200 字摘要
- 提取 3-5 个主题词
- 输出标准化 JSON 结构，方便后续分类统计

### 5.4 全量主题分类与统计

#### 功能描述
对所有视频提取出的主题做归并，形成可展示的主题集合，并统计每个主题下的视频数量与热视频表现。

#### 输出结果
- 主题列表
- 每个主题的视频数
- 每个主题的平均热度
- 每个主题的总热度
- 每个主题的代表视频

### 5.5 定量分析

#### 第一版统计项
- 视频总数
- 平均播放量
- 平均点赞量
- 平均投币量
- 平均收藏量
- 各主题视频数占比
- 各主题热度总量
- 各主题平均互动率

#### 第二版可扩展项
- 发布时间分布
- UP 主集中度
- 新热视频识别
- 主题时间趋势
- 标题关键词共现分析

### 5.6 前端展示

#### 页面建议
1. 任务创建页
2. 任务列表页
3. 任务详情页
4. 视频结果列表页
5. 主题分析看板页

#### 页面展示内容
- 任务执行状态与进度
- 视频列表与基础筛选
- 单视频 AI 摘要
- 主题分类统计图表
- 主题热度排行
- 数据导出按钮

## 6. 非功能要求

### 6.1 合规性
- 采用正规爬虫方式，优先访问公开可见页面和公开接口
- 严格遵守目标站点的服务条款、访问频率限制和 robots 约束
- 不进行账号破解、验证码对抗、登录态滥用、攻击性绕过
- 对访问频率、并发数、重试次数做明确上限控制

### 6.2 稳定性
- 单任务失败可重试
- 页面结构变化后具备快速修复能力
- 关键步骤有日志和异常记录

### 6.3 可扩展性
- 爬虫、AI、统计、前端展示模块解耦
- 支持未来扩展到多关键词、多平台、多模型

### 6.4 可维护性
- 原始采集数据、清洗数据、AI 结果分层存储
- 每个任务可追踪
- 重要参数可配置

## 7. 业务流程

## 7.1 主流程
1. 用户输入关键词并提交任务
2. 后端创建采集任务并进入队列
3. 爬虫模块抓取搜索结果和视频详情
4. 数据清洗与去重
5. AI 模块对单视频文本做总结与主题提取
6. AI/规则模块对全量主题做归并分类
7. 统计模块生成主题热度分析结果
8. 前端展示结果并支持导出

## 9. 推荐技术栈

### 9.1 前端技术栈
- `Vue 3`
- `TypeScript`
- `Vite`
- `Pinia`
- `Vue Router`
- `Element Plus` 或 `Naive UI`
- `ECharts`

选择原因：
- 上手快，适合做简洁清晰的后台型页面
- 组件生态成熟，图表接入简单
- 便于后续扩展任务管理和分析看板

### 9.2 后端技术栈
- `Python 3.11+`
- `FastAPI`
- `Pydantic`
- `SQLAlchemy 2.x`
- `Alembic`
- `Uvicorn`

选择原因：
- Python 生态适合爬虫、NLP、数据分析
- FastAPI 开发速度快，接口文档清晰
- 类型提示和数据校验能力强，适合前后端联调

### 9.3 爬虫技术栈
- `httpx`：请求抓取
- `Playwright`：处理动态页面和兜底渲染
- `selectolax` 或 `parsel`：页面解析
- `tenacity`：失败重试

建议策略：
- 优先使用轻量 HTTP 请求抓取公开数据
- 在页面动态渲染或字段缺失时，用 Playwright 兜底
- 统一封装请求头、Cookie、限速、代理策略

### 9.4 任务与缓存
- `Redis`
- `Celery` 或 `Dramatiq`

建议：
- 如果希望生态成熟，选 `Celery`
- 如果希望更轻量、开发更顺手，选 `Dramatiq`

本项目第一版建议：`Redis + Celery`

### 9.5 数据存储
- `PostgreSQL`：主业务库
- 可选 `MinIO / 本地文件`：保存原始响应和导出文件

表设计建议：
- `crawl_task`
- `video`
- `video_metric_snapshot`
- `video_text_content`
- `ai_summary`
- `topic_cluster`
- `topic_video_relation`

### 9.6 AI 与数据分析
- 大模型接口：兼容 `OpenAI API` 协议或国内兼容接口
- `pandas`：统计分析
- `jieba` 或 `pkuseg`：中文分词辅助
- 可选 `scikit-learn`：主题聚类/相似度计算

建议实现：
- 先用 LLM 做“摘要 + 主题词 + 一级主题”抽取
- 再用规则归并或聚类做全量主题分类
- 输出结构化 JSON，避免纯文本难以复用

### 9.7 部署与工程化
- `Docker`
- `Docker Compose`
- `Nginx`
- `Loguru` 或标准 `logging`

## 10. 爬虫合规与 IP 策略设计

### 10.1 正规爬虫原则
- 只采集公开可访问内容
- 不突破登录边界和访问控制
- 控制采样规模和请求频率
- 明确声明任务仅用于研究、分析或内部使用

### 10.2 IP 使用建议

#### 方案 A：先用自有 IP + 休眠策略
适用于第一版验证，成本最低。

建议配置：
- 单任务串行或低并发
- 每次请求随机休眠 `1.5s - 5s`
- 分页之间增加更长间隔
- 对失败请求指数退避
- 为高频详情页抓取设置总量上限

#### 方案 B：高质量代理 IP
适用于采集频率提升后。

建议要求：
- 住宅代理或高质量静态代理
- 独立出口稳定
- 支持地区切换
- 支持并发量统计与失败率监控

不建议：
- 使用来源不明的廉价代理池
- 高频切换异常 IP
- 试图绕过风控的激进策略

### 10.3 风控保护设计
- User-Agent 池但保持真实、稳定
- 请求速率限制器
- 任务级并发限制
- 失败熔断
- 代理健康检查
- 请求日志审计

## 15. 结论

该项目适合按“先采集、再总结、后分析”的节奏推进。第一版建议重点解决三件事：
- 用合规、低频、可控的方式稳定采到目标视频数据
- 用统一结构保存原始数据与 AI 结果
- 用最简单的前端把任务结果和主题统计清晰展示出来

如果按以上方案推进，项目具备较好的可落地性，也为后续扩展到更多统计维度、更多平台或更复杂的 AI 分析留足空间。
