## 2026-04-16：启动器以“页面可用优先”，后台组件继续异步升温

### 决策

本地开发启动器的成功判定从“前端、后端、Worker 全部 ready 后再返回”调整为：
- 前端页面可访问
- 后端 API 可访问

Worker 保持后台继续升温；若尚未 ready，则给出警告而不是阻塞启动器完成。

### 原因

- 用户体感上的“启动慢”主要来自启动器把 UI 可用时间和 Worker 完整就绪时间绑定。
- 实际上前端页面与后端 API 通常可以更早提供可操作界面。
- Worker 在多数情况下会继续很快完成就绪，不需要把浏览器打开动作强行延后。

### 影响

- `scripts/start-dev.ps1` 的等待逻辑更符合“先可用、后完全就绪”的开发体验。
- 当 Worker 尚未 ready 时，用户仍能进入页面，但会收到明确提示。

## 2026-04-16：最新指标快照查询必须先按 task_id 收敛

### 决策

任务视频列表、主题分析、报告生成等依赖“每个视频最新指标快照”的查询，统一改为先按 `task_id` 过滤，再进行窗口排序和关联。

### 原因

- 旧实现会先对全库 `video_metric_snapshot` 做 `row_number()` 排序，再在外层按任务过滤。
- 当历史任务和快照累积后，这类查询会成为视频列表、主题分析和报告生成的主要性能瓶颈。

### 影响

- `backend/app/services/task_result_service.py`
- `backend/app/services/statistics_service.py`

相关接口在大任务和多任务累计场景下会更稳定，查询成本随单任务数据量而不是全库快照量增长。

## 2026-04-16：主题分析内部 Top N 视频读取改为 SQL 层限流

### 决策

主题分析内部获取 Top N 视频快照时，不再先读取整任务视频结果再切片，而是在查询阶段直接加 `limit`。

### 原因

- 报告和分析只需要少量头部样本时，读取全量结果属于无效 IO 和对象构建。

### 影响

- 主题分析和报告生成的头部样本读取成本降低。
